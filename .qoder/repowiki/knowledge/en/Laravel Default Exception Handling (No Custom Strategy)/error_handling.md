This repository does not implement a custom error-handling strategy. There is no `app/Exceptions` directory and no custom exception classes, sentinel errors, or domain-specific error types defined in the codebase. Error handling relies entirely on Laravel's default framework behavior: uncaught exceptions are rendered via Blade error views under `resources/views/errors/`, HTTP validation failures are handled by Laravel's built-in request validation pipeline, and authorization failures are surfaced through Spatie's role/permission middleware. Controllers and services do not define or throw custom exceptions; business-rule violations appear to be returned as validation responses or view redirects rather than explicit error objects. No global exception handler overrides exist, so logging, response shaping, and API error formatting follow Laravel's out-of-the-box conventions.
