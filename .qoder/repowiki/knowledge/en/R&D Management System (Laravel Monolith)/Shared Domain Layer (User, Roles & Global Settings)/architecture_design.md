Four loosely coupled files that together form the cross-cutting identity and configuration layer:
- `app/Models/User.php` — Eloquent `Authenticatable` augmented with Spatie `HasRoles` trait; defines fillable/hidden attributes via PHP 8 attributes (`#[Fillable]`, `#[Hidden]`) and casts (`email_verified_at`, `password => hashed`). Owns hasMany relationships to `Formula`, `TrialRm`, `TrialPm` on `created_by`.
- `app/Http/Middleware/RoleMiddleware.php` — Request-level gatekeeper that delegates authorization to `$user->hasRole(...)` from Spatie; redirects unauthenticated users to `login` and unauthorized ones to `dashboard` with a flash error.
- `app/Providers/AppServiceProvider.php` — Central wiring point: `register()` requires the global helper file and binds three services as singletons (`FormulaService`, `TrialRmService`, `TrialPmService`); `boot()` registers `Gate::policy` for the three models plus an implicit Superadmin bypass, and composes `layouts.app` with a role-scoped pending-approval notification count.
- `app/Helpers/setting.php` — Standalone procedural helper exposing a `setting($key, $default = null)` function that reads from `App\Models\Setting`; loaded lazily via `require_once` in `AppServiceProvider::register()` so it is available globally without autoloading.

Dependency direction is one-way outward: this module depends on `Spatie\Permission`, other Eloquent models, policies, and services, but nothing in those layers imports back into these four files. The middleware and policy/Gate setup are the explicit boundary between authentication and business logic.
